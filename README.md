# DevOS

## The DevSplit Operating System

> **Build consistently. Think systematically. Continuously improve.**

---

## What is DevOS?

DevOS (DevSplit Operating System) is the institutional knowledge base and operating framework that defines how DevSplit thinks, makes decisions, builds software, delivers services, and collaborates.

It is designed to be used by both **human team members** and **AI agents**, providing a shared source of truth for engineering standards, business processes, technical knowledge, and organizational best practices.

Rather than relying on individual experience or isolated prompts, DevOS captures DevSplit's accumulated knowledge in a structured, reusable, and continuously evolving system.

---

# Vision

To create an operating system that enables humans and AI to work together consistently, efficiently, and according to the same high standards.

Every decision, implementation, review, and recommendation should reflect DevSplit's principles, regardless of who performs the work.

---

# Goals

DevOS exists to:

* Standardize how work is performed.
* Preserve institutional knowledge.
* Improve engineering quality.
* Reduce repetitive decision-making.
* Accelerate onboarding.
* Improve collaboration between specialists.
* Enable AI agents to work consistently.
* Continuously improve through real project experience.

---

# Core Principles

DevOS is built around several fundamental principles.

* Solve the business problem, not just the technical request.
* Understand before implementing.
* Reuse before rewriting.
* Simplicity is preferred over unnecessary complexity.
* Protect existing functionality.
* Never fix one problem by creating another.
* Quality is preferred over speed.
* Document important knowledge once and reuse it everywhere.
* Continuously improve based on real-world experience.

---

# Repository Structure

```
DevOS/

constitution/
    Company-wide principles and standards.

knowledge/
    Technology, business, and domain-specific knowledge.

departments/
    Organizational structure and specialist roles.

playbooks/
    Standard operating procedures for recurring work.

checklists/
    Quality assurance and verification lists.

templates/
    Reusable document and project templates.

examples/
    Practical reference implementations.
```

Each directory has a distinct purpose and should avoid duplicating information found elsewhere.

---

# Architecture

DevOS separates knowledge into different layers.

```
Company Constitution
        │
Engineering & Business Standards
        │
Technology Knowledge
        │
Departments
        │
Roles
        │
Playbooks
        │
Checklists
        │
Projects
```

Each layer builds upon the previous one while maintaining clear responsibilities.

---

# Documentation Philosophy

DevOS is intentionally modular.

Small, focused documents are preferred over large monolithic documents.

Each document should answer a specific question and should remain concise enough to understand quickly.

If a document becomes too large, it should be divided into smaller documents.

---

# How Humans Should Use DevOS

Team members should use DevOS as the primary reference for:

* Company standards
* Engineering decisions
* Technology guidelines
* Project workflows
* Best practices
* Quality expectations

When uncertain, consult DevOS before introducing new patterns or processes.

---

# How AI Agents Should Use DevOS

AI agents should treat DevOS as their primary source of organizational knowledge.

Rather than loading the entire repository, agents should load only the documents relevant to the current task.

For example:

### Software Development

```
Company Constitution
Engineering Standards
Laravel Handbook
Feature Development Playbook
Code Review Guide
Relevant Project Documentation
```

### SEO

```
Company Constitution
Marketing Standards
SEO Knowledge
SEO Playbooks
Relevant Project Documentation
```

### Research

```
Company Constitution
Research Standards
Relevant Knowledge Documents
```

Loading only relevant documentation improves context quality and reduces unnecessary information.

---

# Source of Truth

DevOS is the primary source of truth for:

* Engineering standards
* Organizational processes
* Technical decisions
* Operational workflows
* Internal best practices

When conflicts arise, the following order of precedence applies:

1. Company Constitution
2. Organizational Standards
3. Technology Handbooks
4. Department Documentation
5. Playbooks
6. Checklists
7. Project Documentation

---

# Continuous Improvement

DevOS is a living system.

It is expected to evolve over time as DevSplit gains new experience, adopts new technologies, improves processes, and learns from real projects.

Documentation should be created because it solves a recurring problem—not simply because documentation could exist.

Practical value is preferred over completeness.

---

# Versioning

DevOS follows semantic versioning.

* Major versions introduce structural or architectural changes.
* Minor versions introduce new knowledge and capabilities.
* Patch versions improve existing documentation.

---

# Guiding Philosophy

DevOS is not a collection of prompts.

It is not a static knowledge base.

It is not merely documentation.

DevOS is the operating system that defines how DevSplit works.

Its purpose is to ensure that every engineer, every AI agent, and every future team member approaches problems with the same principles, follows the same standards, and continuously contributes to improving the organization.

---

**Build consistently. Learn continuously. Improve relentlessly.**
